## AIDD-Spec v0.1とは何か

AIDD-Specは、AIエージェントと人間が同じ説明を読み、同じ完了条件でWeb/モバイルシステムを作るための標準仕様である。

今回のv0.1では、次を定義した。

| 項目 | 内容 |
| --- | --- |
| 目的 | AIエージェントに渡す入力、出力、検証証跡、レビュー記録、学習ログを標準化する |
| 適用範囲 | Web/モバイルアプリ、SaaS MVP、AIエージェント実装、レビュー、検証 |
| 成果物モデル | Product / Requirement / Experience / System / Non-Functional / Implementation / Evidence の7層 |
| AI Task Packet | AIエージェントへ渡す標準入力 |
| Verification Evidence | 実行ログ、CI、artifact、スクリーンショットなどの完了証拠 |
| Conformance Levels | L0からL4までの準拠レベル |
| AIDD Control Plane | その流れをSaaS化する方向性 |

## 一番大事な考え方

AIDD-Specは「AIに良い感じにお願いするテンプレート」ではない。

AIに渡す前に、少なくとも次を明確にする。

- 何を作るか
- 何を作らないか
- どの状態を持つか
- どの失敗状態を持つか
- どのファイルを出すか
- どのコマンドで検証するか
- どの証跡を残すか
- 人間が何をレビューするか
- 次回どの学びを戻すか

これを1つの最小単位にしたものが、AI Task Packetである。

## Conformance Levels

v0.1では準拠度を5段階にした。

| Level | 名前 | 条件 |
| --- | --- | --- |
| L0 | Vibe Output | AIが生成しただけ。検証証拠なし |
| L1 | Buildable | 構文/型/buildなど最低限が通る |
| L2 | Testable | unit/E2Eまたは静的監査で主要条件を検証できる |
| L3 | Contract-Verified | 状態/失敗/API/mockを契約として検証できる |
| L4 | CI-Evidenced | CI success、artifact、review、learning logが揃う |

WatchFlow/StudyStream/DrillSwipeのような公開実験はL4を目標にする。

一方、ライブ配信中に誰でも追える最初の実演では、まずL2を狙うのがよい。重すぎると視聴者が追えないからだ。

## 今回の実演: IssueBrief Lite

AIDD-Spec v0.1に沿った入力を作り、Codexに渡して、本当に期待したものが出るかを試した。

実験ディレクトリ:

```text
experiments/aidd-spec-v0.1-validation-001/
```

AI Task Packet:

```text
experiments/aidd-spec-v0.1-validation-001/AI_TASK_PACKET.md
```

Codexへ渡したPrompt:

```text
experiments/aidd-spec-v0.1-validation-001/CODEX_PROMPT.md
```

作らせたものは `IssueBrief Lite` という静的Webアプリである。

目的は、AIに開発タスクを渡す前に、目的・非ゴール・状態・検証条件を1画面で整理すること。

## なぜLite実験にしたか

ここでいきなりNext.js + CI + 3ブラウザまでやると、ライブ配信や読者の再現には重い。

まず確認したかったのは、AIDD-Specに沿った入力を渡すことで、AIが次を守れるかである。

- 指定ファイルを作る
- 日本語UIにする
- empty/loading/success/error/offline/timeoutを入れる
- form labelやaria-liveを入れる
- 外部通信をしない
- 静的検査コマンドを用意する
- review-recordとlearning-logを残す

つまり、今回はL2 Liteの検証である。

## Codexに渡した入力の要点

AI Task Packetでは、次を明示した。

| 入力項目 | 意味 |
| --- | --- |
| Product Brief | IssueBrief Liteが何のためのアプリか |
| Non-goals | API送信、ログイン、DB保存、frameworkを作らない |
| Scope | 出力すべきファイルパス |
| State Contract | empty/loading/success/error/offline/timeout |
| Failure Contract | API未接続、ログイン不要、課金非ゴール、mediaなし |
| Accessibility Contract | label、keyboard、aria-live、ul/li、focus |
| Quality Gates | `npm run lint:static` と `npm run test:contract` |
| Verification Evidence | ログ、review record、learning log |

## Codexの出力

Codexは指定通り、次のファイルを作った。

```text
generated-repo/app.js
generated-repo/docs/learning-log.md
generated-repo/docs/product-brief.md
generated-repo/docs/review-record.md
generated-repo/index.html
generated-repo/package.json
generated-repo/scripts/static-contract-check.mjs
generated-repo/styles.css
```

この時点では、まだ「成功」とは言わない。

AIDD-Specでは、AIの自己申告ではなく、Verification Evidenceを見る。

## 今回言えること / 言えないこと

言えること:

- AIDD-Spec v0.1に沿ったAI Task Packetを渡すと、Codexは期待した出力構造をかなり正確に作れた
- Non-goalsにより、外部API、ログイン、DB、frameworkへのscope creepを防げた
- State Contractにより、empty/loading/success/error/offline/timeoutが出力に入った
- Quality Gatesを指定したため、検証コマンドが生成され、実行できた
- Verification Evidenceとしてログとreview/learning文書を残せた

言えないこと:

- AIDD-Specが完成した業界標準である、とはまだ言えない
- どんなAIエージェントでも同じ品質を出せる、とはまだ言えない
- L4レベルのCI/artifactまでこのLite実験で確認した、とは言えない
- SaaS MVPが完成した、とは言えない

今回言えるのは、**v0.1 draftとして、仕様→入力→AI実行→検証証跡の最小ループが回った** ということだ。

## AIDD Control Planeにする

次に作るべきSaaSは、AIDD Control Planeである。

これはAIコーディングエージェントそのものではない。

主価値は、次の流れを誰でも辿れるようにすることにある。

```text
Product Brief
  -> AI Task Packet
  -> Agent Runbook
  -> Verification Evidence
  -> Review Record
  -> Learning Log
  -> Spec Improvement
```

MVPで作る機能はこうなる。

| 機能 | 目的 |
| --- | --- |
| Product Brief Builder | 何を作るかを短く固定する |
| AI Task Packet Builder | AIへ渡す入力を作る |
| Contract Checker | 必須項目の不足を検出する |
| Agent Runbook Generator | Codex等へ渡すコマンドを生成する |
| Evidence Collector | 実行ログとartifactを紐づける |
| Review Record | 結果を採点する |
| Learning Log | 次回改善点を残す |

設計ファイル:

```text
standards/aidd-control-plane-mvp-v0.1.md
```

## ライブ配信について

ユーザーの要求は「そのワークフローを誰でも辿れるSaaSを開発していく様子をライブ配信」だった。

ここは制約を明確にしておく。

Hermesだけでは、外部配信プラットフォームで実際に配信開始することはできない。配信先アカウント、OBS、ストリームキー、画面共有権限が必要だからだ。

その代わり、配信でそのまま辿れるRunbookを作った。

```text
standards/live-build-runbook-001.md
```

Runbookには次を入れた。

- 配信タイトル案
- 配信の目的
- 配信前チェック
- AIDD-Specを見せる順序
- AI Task Packetを見せる順序
- Codex実行コマンド
- 検証コマンド
- SaaS MVPとして何を作るか
- 次回に作る画面

## 配信で見せる流れ

配信では、以下の順に見せる。

```text
1. AIDD-Spec v0.1を見る
2. AI Task Packetを見る
3. Codexへ渡す
4. 出力されたIssueBrief Liteを見る
5. npm run lint:static / test:contractを実行する
6. Verification Evidenceを見る
7. AIDD Control Plane MVPの設計を見る
8. 次回、SaaSの最初の画面を作る
```

ここまでで、AIDD-Specの考え方を「記事上の概念」ではなく「実行できる流れ」として見せられる。

## 次にやるべきこと

次回は、AIDD Control Plane MVP自体をNext.jsで作り始める。

最初のAI Task Packetは次の範囲に絞る。

```text
/projects/new
/packets/new
/packets/:id/runbook
/packets/:id/review
```

最初の合格条件は次でよい。

- Product Briefを入力できる
- AI Task Packet YAMLを生成できる
- 必須項目不足を表示できる
- Codexへ渡すRunbookを生成できる
- Review RecordとLearning Logの雛形を作れる

これはまだSaaS完成ではない。

しかし、AIDD-Specに沿った開発ワークフローを、SaaSの画面に落とし込む最初の一歩になる。
